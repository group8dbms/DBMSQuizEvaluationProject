Overall Backend Request Flow

Client / Frontend / Postman | v app.js | v attachUser middleware | v Route match (auth/users/exams/...) | v requireAuth / requireRole | v Controller layer | v Service layer | v Supabase Auth / Database | v JSON response

1. Auth Module

Files: routes/authRoutes.js, controllers/authController.js, services/authService.js

APIs:

  • POST /api/auth/signup
  • POST /api/auth/login
  • GET /api/auth/me

Responsibilities:

  • Create user accounts through Supabase Auth.
  • Log users in and return session/token information.
  • Map Supabase auth users to the app-level users table.
  • Expose the currently logged-in user.
  • Restrict public signup to student role.
Client | v authRoutes | v authController | v authService |---------------------> Supabase Auth | +---------------------> usersService ---> users table | v JSON response

2. Users Module

Files: routes/usersRoutes.js, controllers/usersController.js, services/usersService.js

APIs:

  • GET /api/users
  • POST /api/users

Responsibilities:

  • List application users.
  • Create user profile rows in the users table.
  • Find users by auth_user_id.
  • Upsert user profiles after signup.
  • Store app-level role information.
Client | v usersRoutes | v requireRole | v usersController | v usersService | v users table

3. Exams Module

Files: routes/examsRoutes.js, controllers/examsController.js, services/examsService.js

APIs:

  • GET /api/exams
  • GET /api/exams/:id
  • POST /api/exams
  • POST /api/exams/:id/questions

Responsibilities:

  • Fetch exams with their questions.
  • Create new exams.
  • Add questions to existing exams.
  • Check whether an exam is active based on start and end times.
Client | v examsRoutes | v requireAuth / requireRole | v examsController | +--> validate title/start/end | v examsService |------------> exams table | +------------> questions table | v JSON response

4. Submissions Module

Files: routes/submissionsRoutes.js, controllers/submissionsController.js, services/submissionsService.js

APIs:

  • POST /api/submissions/start
  • POST /api/submissions/:id/save
  • POST /api/submissions/:id/submit
  • GET /api/submissions/:id

Responsibilities:

  • Start a submission for a student.
  • Prevent duplicate submissions for the same student/exam pair.
  • Save draft answers.
  • Submit final answers.
  • Generate final_hash at final submit time.
  • Ensure students can access only their own submissions.
  • Ensure submissions only happen while the exam is active.
Student request | v submissionsRoutes | v requireRole(student) / requireAuth | v submissionsController | +--> check exam exists +--> check exam active +--> check ownership | v submissionsService | +--> submissions table +--> generate final_hash on submit | v JSON response

5. Integrity Logs Module

Files: routes/integrityLogsRoutes.js, controllers/integrityLogsController.js, services/integrityLogsService.js

APIs:

  • POST /api/integrity-logs
  • GET /api/submissions/:submissionId/integrity-logs

Responsibilities:

  • Store integrity monitoring events.
  • Ensure logs belong to valid submissions.
  • Prevent students from logging events on someone else’s submission.
  • Expose integrity history to authorized reviewers.
Client | v integrityLogsRoutes | v requireRole | v integrityLogsController | +--> check submission exists +--> check ownership if student | v integrityLogsService | v integrity_logs table

6. Cases Module

Files: routes/casesRoutes.js, controllers/casesController.js, services/casesService.js

APIs:

  • GET /api/cases
  • POST /api/cases
  • PATCH /api/cases/:id

Responsibilities:

  • Open investigation/review cases.
  • Prevent duplicate case creation for one submission.
  • List existing cases.
  • Update case status and verdict.
  • Set resolved_at automatically when resolved.
Admin / Proctor request | v casesRoutes | v requireRole(admin, proctor) | v casesController | +--> check submission exists +--> check if case already exists | v casesService | v cases table

7. Middleware Module

Files: middleware/attachUser.js, middleware/requireAuth.js, middleware/requireRole.js

APIs: used internally by protected route modules.

Responsibilities:

  • Read the Bearer token from the request.
  • Ask Supabase who the token belongs to.
  • Load the matching user profile from the users table.
  • Populate req.user.
  • Enforce authentication and role-based access control.
Incoming request | v attachUser | +--> read Authorization header +--> authClient.auth.getUser(token) +--> usersService.findUserByAuthUserId(...) | v req.user | v requireAuth / requireRole | v allow or reject

8. Config Module

Files: config/env.js

APIs: none, internal only.

Responsibilities:

  • Load the root .env file.
  • Validate required environment variables.
  • Export config values to the rest of the app.
Server startup | v env.js | +--> load .env +--> check required keys | v export config

9. Database Client Module

Files: db/supabase.js

APIs: none, internal only.

Responsibilities:

  • Create the service-role Supabase client.
  • Create the anon-key auth client.
  • Provide reusable clients to all services.
Service layer | v supabase.js | +--> service-role client ---> database operations | +--> anon auth client -----> auth operations

10. HTTP Utilities Module

Files: utils/http.js

APIs: none, helper-only.

Responsibilities:

  • Keep controller responses consistent.
  • Provide shared helpers for 400, 401, 403, 404, and 500.
Controller | v http.js helper | v standard JSON error/response

Final Summary

This backend follows a consistent pattern across all business areas:

  • Routes define the URL structure.
  • Middleware verifies identity and permissions.
  • Controllers validate request data and enforce rules.
  • Services handle database and Supabase interaction.
  • Supabase stores both auth identities and business data.
